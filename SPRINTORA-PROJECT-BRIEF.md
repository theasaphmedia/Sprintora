# Sprintora — Project Brief (for continuing chat sessions)

Last updated: 2026-07-13 (Paystack billing session — replaces the same-day Stripe attempt)

## What this is

Sprintora is a project management SaaS being built toward a real pilot: a company will subscribe and have their team use it. The end goal is to compete with tools like Asana, Monday.com, ClickUp, and Linear.

**Stated plainly, carried over from earlier in this project:** software alone doesn't make a company competitive on a "world stage." That requires capital, a team, a legal entity, a sales/support org, and years of go-to-market execution — none of which exist yet. What has been built is a legitimate technical MVP foundation, not a company. Every session should keep separating "the app works" from "the business exists."

## Name history (why it's called Sprintora)

Started as "Waypoint" → found a direct same-category collision (waypoint.software, an existing PM tool) → tried "Ontrail" → also collided (ontrail.app, a task manager) → settled on **Sprintora**, which had no exact-match collision in casual web searches. Important caveat: casual Googling is not real trademark clearance. Before any full public launch, this still needs an actual USPTO/EUIPO search or a lawyer — not more searching from an AI.

## Competitive research grounding the product's positioning

Sourced via web search during this project (not assumption):
- **Asana**: recurring complaint about notification overload in large workspaces.
- **ClickUp**: steep learning curve ("onboarding my team is taking months"); lag/slow load times in large workspaces (tens of thousands of items).
- **General**: per-seat pricing gets painful as teams scale; user adoption/engagement matters more than raw feature count.

Sprintora's marketing and product decisions lean into these three gaps: staying fast at scale, fast onboarding, and sane notification defaults — not feature-count parity with the big players.

## Stack and infrastructure

- **Frontend/backend**: Next.js 14 (App Router, plain JavaScript, not TypeScript)
- **Auth + database**: Firebase — project ID `sprintora-cda3a` (free Spark plan), Firestore default database in region `nam5`. Auth providers: Email/Password and Google. Email verification is live (see Tier 1 below).
- **Hosting**: Vercel — project name `sprintora`, team `taiglobal`. Production URL: `https://sprintora-taiglobal.vercel.app` (aliases: `sprintora-nine.vercel.app`, `sprintora-theasaphmedia-taiglobal.vercel.app`). Deployment Protection is off, so it's genuinely public.
- **Version control**: a real git repo now backs this, pushed to GitHub at `github.com/theasaphmedia/Sprintora` (public repo), connected to Vercel's Git integration — every `git push` to `main` triggers an automatic production deploy. This did NOT exist as of the last brief update; deploys used to go straight from local file contents with no history. Note: the sandbox this AI runs in cannot execute git commands against the mounted project folder (FUSE filesystem restriction) — the human running the session has to run `git add / commit / push` themselves each time; the AI can prepare commit messages and file changes but not push.
- **Outbound email**: Gmail SMTP via `nodemailer`, authenticated with a Google App Password (env vars `GMAIL_USER` / `GMAIL_APP_PASSWORD` in Vercel). This is a **deliberate free-tier workaround, not the long-term answer** — Resend (the originally planned provider) was tried first but its free/sandbox tier only delivers to the account owner's own registered email address without a verified sending domain, which was a dead end without buying a domain. Gmail SMTP has no such restriction and needs no domain, but caps at 500 emails/day and sends from a personal Gmail address rather than a branded one. Revisit this (buy a domain, verify it with Resend or similar) once either volume or professionalism actually requires it.
- **Source code location**: `C:\Users\USER\Documents\AI\sprintora-app` — this is the real, current source, and is now a live git working directory (see above).
- **Scheduled jobs**: Vercel Cron, configured via `vercel.json` in the repo root (committed, not a dashboard-only setting — Vercel reads it on every deploy). Currently one job: `/api/cron/due-soon`, daily at 13:00 UTC. Requires a `CRON_SECRET` env var (see "New env vars required this session" below) — Vercel automatically sends it as the `Authorization: Bearer <value>` header on every cron-triggered request, which is how the route tells a real scheduled invocation apart from a public hit on the URL.
- **Firebase Admin SDK**: added in the email-notifications session, originally for just one route (`/api/cron/due-soon`). Now also used by the billing routes (`/api/start-trial`, `/api/create-checkout-session`, `/api/create-portal-session`, `/api/webhooks/paystack`, `/api/cron/trial-expiry`) — those still verify the caller's Firebase ID token first (proving *who* is asking) and then use the Admin SDK only to read/write that exact verified uid's own user doc, never anyone else's. Every route that doesn't need this stays on plain ID-token verification (`lib/serverAuth.js`) instead. Requires a `FIREBASE_SERVICE_ACCOUNT_JSON` env var.
- **Payments/billing**: **Paystack, not Stripe.** Stripe was built first this session, then discovered to not support Nigeria as a merchant country at all (confirmed via docs.stripe.com/payments/countries/nigeria — Nigeria has local *payment method* support for Stripe merchants elsewhere, but you cannot create a Stripe account with Nigerian business/bank details). Everything Stripe-specific was torn out and rebuilt against Paystack (Nigerian, Stripe-owned but a completely separate product/API — same-day pivot, not a gradual migration). One flat-rate Pro plan, no per-seat pricing, 14-day free trial. **The trial deliberately collects zero payment info** — Paystack has no clean "authorize a card now, charge later" primitive for Nigerian cards the way Stripe does (their Preauthorization API, which does exactly that, is South-Africa-only/ZAR-only — confirmed via their docs), and an earlier design that worked around this with a small charge-then-refund trick was scrapped in favor of a simpler, more honest approach at the user's explicit request: `/api/start-trial` is a pure Firestore write (no Paystack call at all) that grants Pro access immediately; a card is only ever requested if/when the user adds one, via `/api/create-checkout-session`, which now does a normal, real, immediate-charge Paystack subscription (no more nominal-charge tokenization hack). A new daily cron (`/api/cron/trial-expiry`) reverts anyone whose trial ran out without a card back to the free plan. Requires two new env vars: `PAYSTACK_SECRET_KEY`, `PAYSTACK_PLAN_CODE` (see "New setup required this session" below).
- **Also in this folder**: `sprintora-firestore-rules.txt` (kept in sync with what's published to Firebase console — **always re-publish through the Firebase console's Rules tab after editing this file locally; editing the file alone does nothing to the live rules**) and two legal drafts, `Sprintora-Terms-of-Service-DRAFT.docx` / `Sprintora-Privacy-Policy-DRAFT.docx`.

## Data model (Firestore)

- `users/{uid}`: `email`, `displayName`, `plan` (`"beta"` free or `"pro"` paid), `createdAt`, `paystackCustomerCode`, `paystackSubscriptionCode`, `subscriptionStatus` (`"trialing"|"active"|"non-renewing"|"attention"|"cancelled"|"expired"` — mostly mirrors Paystack's own subscription status vocabulary directly, plus `"trialing"`/`"expired"` which are this app's own states for the card-less trial, not Paystack concepts at all), `trialEndsAt` (ISO datetime string, set once when the trial starts, never touched again), `trialUsed` (boolean, permanent — set `true` the moment a trial starts and never reset, specifically so canceling and re-requesting can't grant a second free trial). **`plan`, `paystackCustomerCode`, `paystackSubscriptionCode`, `subscriptionStatus`, `trialEndsAt`, and `trialUsed` are all write-locked to the client** — only server-side routes using the Admin SDK (`/api/start-trial`, the Paystack webhook, the trial-expiry cron), which bypasses these rules entirely, can set them.
- `projects/{projectId}`: `name`, `ownerId`, `memberIds` (array), `roles` (map of `uid` → `"owner"|"member"`), `createdAt`
- `projects/{projectId}/tasks/{taskId}`: `title`, `status` (`"todo"|"in_progress"|"done"`), `createdAt`, `assigneeId` (uid or `null`, new this session), `dueDate` (`"YYYY-MM-DD"` string or `null`, new this session — deliberately a plain date string, not a Firestore Timestamp, since only the calendar day matters, not a precise moment, and it makes the due-soon cron query a simple string equality check instead of a timezone-sensitive range query)
- `projects/{projectId}/tasks/{taskId}/comments/{commentId}`: `text`, `authorId`, `authorName`, `authorEmail`, `createdAt`. New this session — see Tier 2 below. Comments are immutable once posted (no edit) and can only be deleted by their own author (not even the project owner can delete someone else's comment yet — deliberate scope decision, not an oversight; add owner/moderator delete later if actually needed).
- `projects/{projectId}/tasks/{taskId}/activity/{activityId}`: `type` (`"created"|"moved"|"assigned"|"due-date"`), `fromStatus`/`toStatus` (only on `"moved"`), `assigneeId`/`assigneeName` (only on `"assigned"`), `dueDate` (only on `"due-date"`), `actorId`, `actorName`, `actorEmail`, `createdAt`. `assigned`/`due-date` types added this session. System-generated log entries, not user content — immutable and undeletable by anyone, including via the rules. Rendered merged with comments into one chronological timeline in the task detail view (client-side merge/sort — Firestore can't natively order two separate subcollections together).
- `users/{uid}` gained `emailNotifications` (boolean, new this session) — a missing field is treated as `true` everywhere it's read (never explicitly opted out = notifications on). Toggle lives on the Account page.
- `invites/{inviteId}`: `projectId`, `projectName`, `email`, `invitedBy`, `status` (`"pending"|"accepted"|"left"|"removed"`), `createdAt`. The `status` field is a **single current-value snapshot, not a history log** — it's overwritten on every transition (e.g. removed → re-invited flips it straight back to `"accepted"`, erasing the fact it was ever "removed"). A real audit trail would need a different shape (an array or subcollection of timestamped events) — not built, flagged as a possible future need.
  - **Known gap**: any member added via the "already has an account" branch of the invite flow *before this session's fixes* has no invite doc at all (that branch didn't create one until this session). Their `left`/`removed` status can never be tracked retroactively — there's no record to update. Only affects a handful of early test accounts on the `TWN STUDIOS` test project; not a concern for real users invited going forward.

## Known limitations — be honest about these, don't quietly paper over them

- **Gmail SMTP is a workaround** (see Stack section) — 500 emails/day, unbranded sender address, more spam-filter-prone at scale than a verified domain sender. Fine for beta; revisit before real growth. That 500/day cap is now shared across three email types (invites, task assignments, due-soon reminders) instead of just one — still trivial at current pilot scale, but worth remembering if usage ever grows.
- **The `FIREBASE_SERVICE_ACCOUNT_JSON` credential is a single point of failure if it leaks** — it bypasses every Firestore security rule and can read/write the entire database. It started out scoped to exactly one route (`/api/cron/due-soon`); this session it's also used by every billing route (`/api/start-trial`, `/api/create-checkout-session`, `/api/create-portal-session`, `/api/webhooks/paystack`, `/api/cron/trial-expiry`), so its footprint has grown — still needs real care (Vercel env var only, never logged, never committed to the repo), more so now than before.
- **The invite-notification API route (`/api/send-invite`) verifies the requester is signed in, but not that they actually own the project they're inviting to.** Documented in the route's own code comment. Practical risk today: any signed-in Sprintora user could trigger an invite email to an arbitrary address via any project ID, not just one they own. Low severity (no data exposure, just unwanted email), not fixed yet.
- **Invite `status` has no history** — see data-model note above.
- **Untested paths**: password reset ("Forgot password?"), and — now that self-leave/remove exist — re-verify them again after any future rules changes, since they depend on precise rule wording that's easy to accidentally loosen.
- **Nothing is actually gated behind the paid "pro" plan yet.** Trial start, real subscription, webhook, and portal are all fully built this session, but the app doesn't restrict any feature by plan — a free "beta" user and a paid "pro" (trialing or fully subscribed) user currently have identical access to everything. This is a real, open product decision (what does "pro" actually unlock?), not something decided unilaterally while building the payment plumbing — flag it for a real answer before pushing this live to real users.
- **Paystack billing needs two new env vars plus a few one-time Paystack Dashboard steps before it works at all in production** — see "New setup required this session" below. Until then, "Start 14-day free trial" will still work (it never touches Paystack), but "Add payment method" / "Subscribe to Pro" will fail with a visible error.
- **The card-less trial has a real gap: it doesn't verify the person starting it is a distinct human/business.** Because no payment info is ever collected to start a trial, nothing stops one person from creating multiple Sprintora accounts (different emails) to get repeated free trials — `trialUsed` only prevents re-using a trial on the *same* account. Low severity at pilot scale (not worth fraud-prevention infrastructure yet), but a real, known limitation, not an oversight to pretend doesn't exist.
- **Refund/dispute path for the trial doesn't exist because there's nothing to refund** — a deliberate consequence of never charging anything during the trial. If the earlier charge-then-refund design (scrapped this session) had shipped instead, refund failures would have been a real operational risk; worth remembering if a "verify the card is real" step ever gets added back later.
- No legal entity formed yet. The ToS/Privacy Policy drafts have placeholder fields (`[COMPANY LEGAL NAME]`, `[CONTACT EMAIL]`, governing jurisdiction) and **must be reviewed by an actual lawyer** before any real company relies on them.
- Firebase is on the free Spark plan: hard daily caps (50,000 reads / 20,000 writes per day). Fine for a small pilot; upgrade to the Blaze pay-as-you-go plan before wider usage, otherwise the app can simply stop responding mid-pilot if caps are hit. (User has said: upgrade "when we get there.")
- Still running on the `vercel.app` subdomain, not a purchased custom domain. (User has said: staying on Vercel for now, will purchase a domain once everything else is done.) Buying a domain would also unlock a proper verified-domain email sender, solving two problems at once if it ever happens.
- No error monitoring (Sentry or similar), no automated tests, no rate limiting on the invite endpoint, no Firestore backup schedule. None urgent at current scale; all worth having before a real paying pilot.

## Tier 2 — in progress

Competitive-parity features — expected by anyone evaluating a PM tool, not differentiators on their own, but their absence reads as "unfinished":
- **Comments and activity feed on tasks — built this session, fully done.** Click a task card to open a detail modal with a merged, chronologically-ordered timeline: user comments and auto-logged system events ("created this task", "moved this from To Do to In Progress") interleaved together.
- **File attachments on tasks — deliberately deferred, not started.** Requires Firebase Cloud Storage, which as of 2026 requires the paid Blaze plan (removed from the free Spark plan entirely, even for trivial usage). Explicit decision this session: don't upgrade to Blaze speculatively — wait until real users actually ask for file attachments, then decide. A viable alternative surfaced and discussed but not built: Supabase Storage (free tier, no credit card, 1GB) as a second backend alongside Firebase — would need a server-side bridge to verify the Firebase-authenticated user before touching it (same pattern as the Gmail-SMTP invite route), and has its own gotcha (free Supabase *projects* auto-pause after 7 days of no database activity — unconfirmed whether Storage-only usage counts against that). Decide the Firebase-Storage-vs-Supabase-Storage question for real when this becomes an actual, current build item.
- **Search across tasks and projects — built this session, fully done.** A search box at the top of the dashboard filters projects (already-loaded, client-side substring match on name) and tasks (client-side substring match on title). Task search requires a one-time index build: on first focus/keystroke, `ensureTaskIndex()` loops the user's own projects and fetches each one's `tasks` subcollection via `getDocs`, then caches the combined list in state for the rest of the session. Deliberately **not** a Firestore `collectionGroup("tasks")` query — that would need to span every user's tasks database-wide, and task docs don't store membership info for a security rule to scope it by, so it was rejected as a privacy hole. Clicking a task result deep-links to `/dashboard/{projectId}?task={taskId}`, which the project board page auto-opens via a `useSearchParams`-driven effect (guarded with a `useRef` so it only fires once, not on every task-list update). No Firestore rules changes were needed — it only reads data the existing rules already permit. Not yet tested live by the user as of this brief update.
- **Email notifications (assigned to you, due soon) — built this session, fully done, not yet user-tested.** Two triggers, plus an opt-out, exactly as scoped: (1) **assignment emails** — fire synchronously the moment a task is assigned to someone (task detail modal now has an Assignee dropdown), reusing the Gmail-SMTP path via a new `/api/send-notification` route; skips emailing someone for assigning a task to themselves. (2) **due-tomorrow reminders** — a new Vercel Cron job (`/api/cron/due-soon`, daily) finds every task across every project whose `dueDate` is exactly tomorrow and isn't done, groups by assignee so one person with several tasks due gets one email not several, and sends via the same Gmail SMTP account. Deliberately a fixed "exactly tomorrow" window rather than "due within N days" — each task then matches on exactly one day, so no extra state is needed to avoid re-notifying the same task daily. (3) **opt-out** — an `emailNotifications` checkbox on the Account page gates both triggers; default is on. This whole feature needed **zero Firestore rules changes** (every field it touches was already writable under the existing member/self-only rules) but **does need two new Vercel env vars and one Firebase Console step before it will actually work in production** — see "New setup required this session" below. Not yet tested live: the assignment-email path can be tested immediately by assigning a task to a teammate account; the due-soon path won't run for real until either the daily cron fires or it's manually invoked with the `CRON_SECRET`.

## Tier 3 — differentiation ideas, not urgent build items

- Keep the core differentiators front and center as the product grows: stays fast at scale, fast onboarding, sane notifications, transparent pricing.
- **Flat, non-per-seat pricing via Paystack — built this session (rebuilt from an earlier same-session Stripe attempt — see Stack section for why).** One Pro plan, one flat monthly price (the actual amount is set in the Paystack Dashboard when creating the Plan, not hardcoded — deliberately left as a business decision, and will be NGN-denominated since that's Paystack's native currency for a Nigerian account), 14-day free trial with zero card collection, hosted Paystack Checkout for the real subscribe flow and Paystack's subscription-management page as the closest available Customer-Portal equivalent (narrower than Stripe's — no invoice history, no plan switching, just card update/cancel). Code-complete; needs the Paystack Dashboard setup below before the real-payment path is live (the trial itself works with zero setup, since it never touches Paystack), and — separately — an actual decision about what "pro" unlocks (see Known limitations).
- AI-assisted features (auto-summarized status updates, smart due-date suggestions) — still not started, and on reflection, lower-priority than billing was: neither addresses the specific competitor gaps the original research surfaced (notification overload, onboarding friction, per-seat pricing pain), so they read as feature-count additions rather than the differentiation Sprintora's own positioning argues against competing on.

## Explicit business decisions made so far (don't re-litigate these without new information)

- Legal entity: still deferred — no company exists yet to formally receive payouts into. **Real billing itself is no longer deferred** — built this session at the user's explicit request, superseding the earlier "we can add later" decision. Payments processor: **Paystack, not Stripe** — Stripe doesn't support Nigeria as a merchant country at all (confirmed via their own docs), so the user (Nigeria-based) can't open a Stripe account; Paystack (Nigerian, Stripe-owned but a separate product) was chosen instead after comparing it against Flutterwave, forming a US entity to keep Stripe, and merchant-of-record platforms (Paddle/Lemon Squeezy, whose Nigeria payout support couldn't be confirmed). Payouts land in whatever Nigerian bank account gets connected when the Paystack account is set up — worth being deliberate about how that interacts with "no legal entity yet" before real money moves.
- **Trial design: no card collected at signup.** Explicit user decision this session, overriding an earlier AI-proposed design that charged a small nominal amount and refunded it immediately to tokenize a card for a delayed-first-charge trial. The user's reasoning: "just entering the market" — card-less trial signup lowers friction, matters more than replicating Stripe's exact trial mechanics. Real tradeoff accepted along with this (see Known limitations): no card at signup means no way to prevent the same person from claiming multiple trials across different email addresses.
- Firebase Blaze upgrade: deferred until usage approaches free-tier limits.
- Custom domain: deferred until later; staying on the Vercel subdomain for now.
- International market for v1: English-only, global availability (not multi-language, not EU-specific compliance work) — that was the explicit choice made earlier.
- Membership removal: only the owner can remove other members; any member can remove (leave) themselves; no one can remove someone other than themselves except the owner. Confirmed and built out this session, not just a rules quirk — this is the intended policy.
- Email delivery: staying on free Gmail SMTP for now rather than buying a domain purely to unblock Resend; revisit if/when a custom domain gets purchased anyway.

## New setup required this session (must be done in Firebase/Paystack/Vercel dashboards, not by AI)

**Carried over from earlier sessions, still needed if not already done:**

1. **`FIREBASE_SERVICE_ACCOUNT_JSON`** — Firebase Console → gear icon → Project Settings → Service Accounts tab → "Generate new private key" → downloads a JSON file. Paste the entire file contents as the value of this env var in Vercel (Project → Settings → Environment Variables). **Treat this like a password.**
2. **`CRON_SECRET`** — any random 16+ character string. Vercel automatically sends it as the cron job's auth header.

**New this session — required before real Paystack subscriptions work (the trial itself needs none of this):**

3. **Create a Paystack account** (paystack.com) if one doesn't exist yet, with a Nigerian business/bank setup. Stays in **test mode** until deliberately switched to live — test mode uses fake card numbers, no real money moves.
4. **Create a Plan** in the Paystack Dashboard (Plans → Create Plan), interval "monthly." This is where the actual Naira amount gets decided — deliberately not hardcoded in the code, so this is a real business decision. Copy the resulting plan code (starts `PLN_...`).
5. **`PAYSTACK_SECRET_KEY`** — Paystack Dashboard → Settings → API Keys & Webhooks → Secret Key (starts `sk_test_...` in test mode). Add as a Vercel env var. **Treat like a password** — it can create charges and read customer data. This same key is also used to verify webhook authenticity, so there's no separate webhook-signing-secret env var needed (unlike the earlier Stripe attempt).
6. **`PAYSTACK_PLAN_CODE`** — the plan code from step 4. Add as a Vercel env var.
7. **Configure the webhook endpoint**: Paystack Dashboard → Settings → API Keys & Webhooks → Webhook URL → `https://sprintora-nine.vercel.app/api/webhooks/paystack`. Paystack doesn't let you pick specific event types to send (unlike Stripe) — it sends everything to one URL, and the route itself only acts on the events it recognizes (`subscription.create`, `subscription.not_renew`, `subscription.disable`, `invoice.payment_failed`) and ignores the rest.

After adding both env vars, trigger a fresh deployment — env vars only apply to deployments created after they're added.

**Testing before going live**: click "Start 14-day free trial" first — it needs none of the above, since it's pure Firestore. To test the real-payment path, use Paystack's test card numbers (e.g. `4084 0840 8408 4081`, any future expiry, any CVV, PIN `0000`, OTP `123456`) via "Add payment method" or "Subscribe to Pro" and confirm the Account page updates to show an active subscription within a few seconds. Only switch `PAYSTACK_SECRET_KEY`/`PAYSTACK_PLAN_CODE` to their live-mode equivalents once that loop is confirmed working.

**One-time Firestore index notes**: two separate composite/collection-group indexes may be needed the first time their queries actually run — the due-soon cron's collection-group query on `tasks.dueDate` (from the previous session), and the new trial-expiry cron's compound query on `users` (`subscriptionStatus == "trialing"` AND `trialEndsAt <= now`). Both fail with a `failed-precondition` error containing a direct create-index link on first run if the index doesn't exist yet — check Vercel's logs for `/api/cron/due-soon` and `/api/cron/trial-expiry` after the first scheduled run of each.

**Manual cleanup needed**: the sandbox this AI runs in can't delete files from the mounted project folder (a filesystem restriction), so two now-dead Stripe files were left in place as inert stubs rather than actually removed: `lib/stripe.js` and `app/api/webhooks/stripe/route.js`. Safe to delete both manually:
```
rm lib/stripe.js
rm -rf app/api/webhooks/stripe
```
